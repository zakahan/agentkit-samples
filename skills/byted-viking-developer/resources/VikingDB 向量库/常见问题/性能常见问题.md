1. **向量数据库SDK调用检索有点慢，有什么优化方法吗？**

首先请参考[测试工具V2](https://www.volcengine.com/docs/84313/2179005?lang=zh)检查公网延时。如果是因为公网网络延迟，建议使用私网连接可改善时长，请参考[VikingDB 私网连接方式](https://www.volcengine.com/docs/84313/1254445)。


2. **CPU 使用率受哪些因素的影响？**
* QPS：高 QPS 会使 CPU 使用率升高，因为单位时间内要处理更多任务。
* topk 太大：会增加 CPU 的计算量和排序筛选工作，提高 CPU 使用率。
* dsl 太过复杂：复杂的 DSL 会让 CPU 花费更多的运算资源来解析和执行相应的检索逻辑。



3. **数据插入与查询同时进行时，查询性能会下降吗？**

不会，因为我们是存算分离的架构，因此数据的写入完全不会影响到在线检索。


4. **向量库数据写入效率的限制是多少？**

同步写入限流是 1000 条数据/秒；异步写入限流是10000条数据/秒，对于需向量化的数据还需要考虑token限流情况。


5. **插入、删除或更新数据后，是否需要刷新才能查询到最新数据？**

不需要刷新即可直接查询到数据。



6. **为什么索引创建页面显示的预估 QPS 和实际测下来的 QPS 不符？**

索引创建页面中显示的QPS是根据CU估算出来的值。然而，单次请求占用的CPU和请求参数（limit值、检索条件等）紧密相关，不同场景下的CPU消耗会有较大的差别，进而导致在相同CU配置条件下，能支持的QPS差别也比较大


7. **如何根据业务场景预估资源使用量？**

目前建议连接产品解决方案或产解进行辅助预估，后续将上线资源自动预估功能。


8. **单个数据集的数据量上限是多少？数据量达到多少量级会导致检索速度变慢？** 

单个数据集数据量没有上限。
百万以上的数据量会对检索速度产生影响，具体影响程度与向量维度、检索条件、索引类型和量化方式等都有关系。


9. **向量库使用时有几个副本对外提供服务，副本如何分配计算资源？**

向量数据库使用时会有至少两个副本提供服务，计算资源在流量上是均分的。


10. **为什么CU未达到上限但是search请求被限流了，返回报错 "Code":"TooManyRequests","message":"请求次数已达上限, 请控制访问频率"？**

可能是因为接口请求不符合预期导致collection 或 index 查询接口被限流，需要检查使用方法。
使用时，index 和 collection 只初始化一次即可，不需要每次检索都重新执行。


11. **调用豆包模型如何正确上量，避免接口调用异常？**

1、上量前需提前通知火山引擎VikingDB相关对接人员，以便做好资源预留准备。
2、按照梯度爬坡逐步提升用量，具体爬坡策略可参考：起始流量200wTPM，每5分钟涨200wTPM
3、以图片数据的token消耗为例，300KB图片大约消耗1000token。实际token以具体数据为准。


12. **调用豆包 embedding vision 模型时遇到限流 / 接口异常，如何处理？**

接口限流多由突发流量触发，如遇到限流，请暂停流量增长并联系火山引擎VikingDB对接人员。
